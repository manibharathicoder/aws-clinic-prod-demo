# AWS Services & Architecture Summary

## 🛠️ AWS Services Used: What & Why

**1. S3 (Simple Storage Service)**
* **What:** Cloud storage buckets.
* **Why:** To store the compiled React frontend files (`index.html`, Javascript, CSS) permanently and cheaply.

**2. CloudFront (CDN)**
* **What:** Global Content Delivery Network.
* **Why:** To serve the frontend quickly from global edge locations, provide free HTTPS (SSL), and act as a secure proxy to connect the frontend and backend together on the exact same domain.

**3. Elastic Beanstalk (EB)**
* **What:** Fully managed application hosting.
* **Why:** To host the Node.js/Express backend easily. It automatically handles the EC2 server setup, load balancing, restarts if the app crashes, and applies OS updates without manual work.

**4. RDS (Relational Database Service)**
* **What:** Managed PostgreSQL database.
* **Why:** To store all clinic data securely. It automatically handles backups, updates, and is isolated in a private network (crucial for HIPAA compliance).

**5. Secrets Manager**
* **What:** Secure credential storage vault.
* **Why:** To store sensitive keys (like database passwords and Stripe keys) so they are never written directly in the code or easily accessible to unauthorized users.

**6. CloudWatch**
* **What:** Monitoring and alerts.
* **Why:** To watch server CPU, database health, and billing. It triggers alarms to notify the team if the app crashes or if the AWS bill gets too high.

# AWS CORS & Mixed Content Fixes (CloudFront Proxy Trick)

This document contains the exact steps and explanations for how we solved the major network errors when deploying the AIM Pediatric Therapy platform to AWS.

## Error 1: "Mixed Content" & "Network Error"
**The Problem:** 
CloudFront gave our frontend a secure `https://` address. Elastic Beanstalk (our backend) was stuck on an insecure `http://` address. Modern browsers (like Chrome) strictly block secure websites from sending data to insecure websites. This is called a **Mixed Content** block.

**The Fix ("The Senior Cloud Engineer Proxy Trick"):**
Instead of buying a custom domain and SSL certificate right now, we hacked CloudFront. We configured CloudFront to act as a reverse proxy for BOTH our frontend and backend.
1. We modified the CloudFront configuration to add Elastic Beanstalk as a second "Origin".
2. We added a Cache Behavior rule: `Any request to /api/* gets sent to Elastic Beanstalk, everything else goes to S3`.
3. Finally, we updated `VITE_API_URL=/api/v1` in `aim-frontend/.env.production` so the React app uses its own secure exact URL.
Chrome now sees HTTPS talking to HTTPS, and the Mixed Content Error is permanently bypassed!

## Error 2: "CORS Blocked"
**The Problem:**
Our Node.js backend formally rejects requests that don't come from allowed origins. Since the frontend now lived at a new CloudFront domain (`https://d7bskk4z93my5.cloudfront.net`), the backend blocked it.

**The Fix:**
We updated the environmental variables on Elastic Beanstalk to explicitly trust the CloudFront domain:
`eb setenv CLIENT_URL=https://d7bskk4z93my5.cloudfront.net`
We also told CloudFront to forward the `Origin` and `Authorization` headers in the `cf-config.json` file so Elastic Beanstalk could read them.

## Error 3: "504 Gateway Timeout"
**The Problem:**
Once CloudFront was set up to proxy the backend, the AWS Firewall (Security Group) completely blocked CloudFront's attempts to connect to the backend because it didn't recognize the incoming traffic. 

**The Fix (Unlock Port 80):**
We found the Security Group ID for Elastic Beanstalk and explicitly told the firewall to allow all TCP traffic on Port 80 so CloudFront could pass through.
Command used:
`aws ec2 authorize-security-group-ingress --group-id sg-0ff51a814ebf9dad8 --protocol tcp --port 80 --cidr 0.0.0.0/0`

## Error 4: S3 "AccessDenied" XML Error
**The Problem:**
S3 buckets block all public access by default. CloudFront was knocking on S3's door to get the `index.html` file, but S3 rejected it.

**The Fix (VIP Keycard):**
We removed the "Block All" safety shield and explicitly added a Bucket Policy allowing public read access (`s3:GetObject`).
Then, we invalidated the CloudFront cache (`aws cloudfront create-invalidation`) so that it fetched the newly unblocked files instead of remembering the old error message.
