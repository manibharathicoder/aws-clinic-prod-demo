# Project Summary – AIM Pediatric Therapy

## Overview
AIM is a pediatric therapy management system consisting of a **Node.js/Express backend API** and a **React + Vite frontend**. The backend serves JSON endpoints for authentication, child/therapist management, and session scheduling. The frontend provides a role‑based UI (Super Admin, Admin, Accounting, Therapist, Family) with protected routes and placeholder pages for future features.

## Tech Stack
- **Backend**: Node.js, Express, PostgreSQL, JWT, Helmet, CORS, dotenv, Winston logger
- **Frontend**: React (v19), Vite (v8), React Router v7, Tailwind CSS v4, Axios
- **Database**: PostgreSQL (AWS RDS) – migrations in `src/migrations/*.sql`
- **Deployment**: AWS Elastic Beanstalk (see `.elasticbeanstalk` config)

## Backend Details
- Entry point: `app.js`
  - Sets security middleware (`helmet`, `cors`), JSON parsers, and mounts routes under `/api/v1/`
  - Health endpoint: `GET /api/v1/health`
  - Global 404 and error handlers
- **Routes** (`src/routes`):
  - `authRoutes.js` – `/login` (POST) and `/me` (GET, JWT protected)
  - `childRoutes.js` – CRUD for children
  - `therapistRoutes.js` – therapist profile & list
  - `sessionRoutes.js` – session creation & retrieval
- **Auth** (`src/middleware/auth.js`): verifies JWT, adds `req.user`
- **Controllers** (`src/controllers`): implement business logic for each entity
- **Database config** (`src/config/db.js`): uses environment vars (`DB_HOST`, `DB_PORT`, etc.) with SSL disabled verification for RDS
- **Scripts** (`package.json`):
  - `start`: runs migrations then `app.js`
  - `dev`: nodemon for hot reload
  - `test`: Jest (placeholder)

## Frontend Details
- Entry point: `src/main.jsx` → renders `<App />`
- **App.jsx** sets up routing with `BrowserRouter`
- **AuthContext** provides authentication state and token handling
- **ProtectedRoute** checks `allowedRoles` against the user’s role decoded from JWT
- **DashboardLayout** supplies a common layout (sidebar, header)
- **Pages**:
  - `LoginPage.jsx` – login form
  - `Unauthorized.jsx` – shown when role check fails
  - Role‑specific dashboards under `pages/<role>/Dashboard.jsx`
  - `ComingSoon` component used for not‑yet‑implemented pages
- **Routing hierarchy** (example):
  - `/superadmin/dashboard` → `SuperAdminDashboard`
  - `/admin/dashboard` → `AdminDashboard`
  - `/accounting/dashboard` → `AccountingDashboard`
  - `/therapist/schedule` → `TherapistSchedule`
  - `/family/sessions` → `FamilySessions`
- **Styling**: Tailwind CSS utilities; minimal custom CSS
- **Dependencies** listed in `package.json`

## Database Migrations
- `001_create_users.sql` – creates users table with role & password hash
- `002_clinic_core.sql` – creates child, therapist, session tables and relationships
- Seed scripts (`seed_users.js`, `seed_clinic.js`) populate demo data

## Deployment & Config
- Elastic Beanstalk configuration lives in `aim-backend/.elasticbeanstalk`
- Environment variables (`.env`, `.env.example`) contain DB credentials and `CLIENT_URL`
- CloudFront and bucket‑policy files for static asset hosting of the frontend (`cf-config.json`, `bucket-policy.json`)

## How It Works (request flow)
1. **Client** loads the React app from the `aim-frontend/dist` folder.
2. On login, the frontend posts credentials to `POST /api/v1/auth/login`.
3. Backend validates, issues a JWT, and returns user info (including role).
4. JWT is stored in local storage; `AuthContext` injects it into subsequent API calls via `Authorization: Bearer <token>`.
5. When navigating to a protected route, `ProtectedRoute` decodes the token, checks `allowedRoles`, and either renders the page or redirects to `/unauthorized`.
6. CRUD operations for children, therapists, and sessions are performed through the respective API endpoints, each guarded by `verifyToken` middleware.
7. All data persists in the PostgreSQL RDS instance via the `pg` pool configured in `db.js`.
