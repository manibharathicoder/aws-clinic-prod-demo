JWT token required on every protected API route — verify in middleware before handler runs2.  Role checked on every route — a therapist cannot access accounting endpoints, ever3.  .env file must be in .gitignore from day one — never commit credentials to GitHub4.  All secrets in AWS Secrets Manager — not in environment variables on the server directly5.  RDS database must have public accessibility OFF — verify this after every deploy6.  Use helmet npm package on Express — sets essential HTTP security headers automatically7.  Input validation on every POST/PUT endpoint — never trust user-submitted data8.  CORS configured to allow only the frontend CloudFront domain — not wildcard (*)9.  All S3 documents accessed via pre-signed URLs — never expose bucket directly10. HTTPS enforced everywhere — HTTP requests must redirect to HTTPSAPI conventionsAll API routes must follow this consistent pattern:// Route structureGET    /api/v1/sessions          # get all (with filters)GET    /api/v1/sessions/:id      # get onePOST   /api/v1/sessions          # createPUT    /api/v1/sessions/:id      # updateDELETE /api/v1/sessions/:id      # delete (soft delete only) // Every route must:// 1. Pass through auth middleware (verifyToken)// 2. Pass through role middleware (requireRole)// 3. Validate input before processing// 4. Local development is faster and safer for testing.Step 1 — Install tools on your machine# Required on your machineNode.js v20 LTS    →  nodejs.orgGit                →  git-scm.comPostgreSQL 15      →  postgresql.orgAWS CLI            →  aws.amazon.com/cliEB CLI             →  pip install awsebcli # Verify everything installednode --version     # should show v20.x.xnpm --version      # should show 10.x.xgit --version      # should show 2.x.xpsql --version     # should show 15.xaws --version      # should show aws-cli/2.x.xeb --version       # should show EB CLI 3.x.xStep 2 — Clone the repositories# Backendgit clone git@github.com:aim-pediatric-therapy/aim-backend.gitcd aim-backendnpm installcp .env.example .env# Edit .env with your local database credentials # Frontendgit clone git@github.com:aim-pediatric-therapy/aim-frontend.gitcd aim-frontendnpm installStep 3 — Set up local database# Create the local databasepsql -U postgresCREATE DATABASE aim_dev;CREATE USER aim_admin WITH PASSWORD 'your_local_password';GRANT ALL PRIVILEGES ON DATABASE aim_dev TO aim_admin;\q # Run migrations (once migration files are created)npm run db:migrate # Run seed data for testingnpm run db:seedStep 4 — Configure AWS CLI on your machine# Configure using your IAM access keysaws configure AWS Access Key ID:     [your dev-fullstack access key]AWS Secret Access Key: [your dev-fullstack secret key]Default region:        ca-central-1Default output format: json # Verify it worksaws sts get-caller-identity# Should show your IAM user account detailsDeploying the backend to Elastic BeanstalkFirst-time setupcd aim-backend # Initialise EB in the projecteb init# Select: ca-central-1# Select: Node.js# Application name: aim-backend# SSH keypair: create new or select existing # Create the dev environment (first time only)eb create aim-backend-dev# This takes 5-10 minutes — AWS is provisioning the server # Set environment variables on the servereb setenv NODE_ENV=production \  DB_HOST=aim-db-dev.xxx.ca-central-1.rds.amazonaws.com \  AWS_REGION=ca-central-1 \  SECRETS_NAME=aim/dev/secrets # Check it is runningeb statuseb open    # opens the URL in browserEvery subsequent deploymentcd aim-backend # Deploy latest codeeb deploy aim-backend-dev # Watch the logs during deploymenteb logs --all # Check health after deployeb healthDeploying the frontend to S3 + CloudFrontcd aim-frontend # Build the React app for productionnpm run build # Upload to S3 (replace bucket name as needed)aws s3 sync build/ s3://aim-frontend-prod --delete # Invalidate CloudFront cache so users see new versionaws cloudfront create-invalidation \  --distribution-id YOUR_DISTRIBUTION_ID \  --paths "/*" # Verify the frontend is live# Open the CloudFront URL in browser# Should show the new versionCloudWatch monitoring setupSet up these alarms after the first deployment.
